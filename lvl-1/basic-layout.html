This is a fairly basic example of how a page should look semantically: Nav at the top, content below. You see that each link doesn't go to a page but just jumps down the screen: That's an "anchor". The href refers to the ID of the element on the page. Same way a style can be applied to an ID with a hash.

As to the layout, let's break down what happened: I put the whole page in an article, "main-content" class. Typically an article wouldn't be the whole page, but in this case we don't have an actual "site", just individual pages. So it makes sense.

That article holds the white box that's centered on the page; it's centered due to the margin and the max-width properties. That's the "auto" margin trick and it's really useful, so keep it in mind. It only works with block elements that have a width which is less than the width of the container.

Within that centered container we have a header, then sections. Within the header we have headlines, followed by a nav element containing a list of nav-items. All of that wraps up and we're on to individual sections of the page.

On the note of that auto margin trick, here's the perfect use: Images. Images are inline elements, but they act like shit that way, so mostly we change them. The format I use is to make it act like a block level element and position it from there. Here's three examples:

Your basic image. It's an inline element, and since it's an image it'll just ignore the container and spill over. Ugly.

Make it act like a block, and tell it what width to be. The width percent is relative to the parent: Since the parent has a certain width, the image will be 75% of that width. It's worth mention: Percentage values always represent width. Even if you set "height: 50%", that 50% value is 50% of the available width, not 50% of the available height. This is a weird quirk that doesn't usually come up, but it goes to show that there's more going on than you might think. It also shows that I know my shit. Gotta flex, son.
